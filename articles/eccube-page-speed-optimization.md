EC-CUBE の PageSpeed Insights を計測したら赤点だった──そんな経験はないでしょうか。「重い気がするけど後回しにしてきた」という状況は、実は**毎日売上を失い続けている**状態です。設定変更だけで速くなる施策も多く、費用をかけずに改善できるケースは少なくありません。

- 読み込みが **1秒 → 3秒** になると、離脱率が **32% 上昇**（Google）
- モバイルで **3秒以上** かかると、**53% のユーザーが離脱**（Google）
- 表示速度が **1秒低下**するごとに、CVR が **20% 以上低下**（Akamai）

本記事では、EC-CUBE 4.3 の構成を踏まえた**表示速度改善の施策**を、インフラ・アプリケーション・フロントエンドの各レイヤーで整理します。

## こんな方に読んでほしい

- EC-CUBE ストアの PageSpeed Insights スコアが低い
- 商品一覧や商品詳細ページの表示が遅いと感じる
- 施策の優先度が分からず何から手をつけるか迷っている

---

## EC-CUBE 4.3 の表示速度に影響する構造的な特徴

改善策に入る前に、EC-CUBE 4.3 の構造的な特徴を把握しておきます。

### リクエストごとに BaseInfo を取得する

EC-CUBE は `TwigInitializeListener` がリクエストのたびに `BaseInfoRepository::get()` を呼び出し、店舗基本情報を Twig グローバル変数に登録します。`$this->initialized` フラグによる重複実行防止はありますが、初回リクエスト時には必ずDBアクセスが発生します。

### 商品一覧は複雑なクエリを実行する

商品一覧（`ProductController::index()`）は `ProductRepository::getQueryBuilderBySearchData()` で複数の JOIN を伴う複雑なクエリを実行します。カテゴリ絞り込み・名前検索・価格ソートが重なると、クエリのコストは急増します。

### デフォルト設定では APCu キャッシュが無効

EC-CUBE の `prod/framework.yaml` には APCu を使ったシステムキャッシュ設定がコメントアウトの状態で存在します。つまり、**デフォルトでは高速なインメモリキャッシュが使われていません**。

---

## 施策1：OPcache を正しく設定する（最優先）

PHP コードのコンパイル結果をメモリに保持する OPcache は、最もコストパフォーマンスの高い施策です。

推奨設定の考え方は以下の通りです。

| 設定項目 | 推奨方針 |
|---------|---------|
| `opcache.enable` | 1（有効化） |
| `opcache.memory_consumption` | 256MB 以上を目安に（デフォルト128MB） |
| `opcache.max_accelerated_files` | プロジェクトの PHP ファイル数より大きく設定 |
| `opcache.validate_timestamps` | 本番では 0（デプロイ時に手動リセット） |
| `opcache.interned_strings_buffer` | 32MB 以上を目安に |

`opcache.validate_timestamps=0` にすると、PHP ファイルの変更がキャッシュに反映されなくなります。デプロイ後は PHP-FPM の再起動か `opcache_reset()` が必要です。

:::message alert
**セキュリティ上の注意：** デプロイ後に OPcache のリセットを忘れると、セキュリティパッチを含むコード修正が本番に反映されないまま稼働し続けます。デプロイスクリプトに OPcache リセット処理を必ず組み込んでください。また、`opcache_reset()` を呼び出す Web エンドポイントを設置する場合は、外部からアクセスできないよう IP 制限等で保護してください。
:::

### OPcache Preload（PHP 7.4 以降）

OPcache Preload を利用するには、`config/preload.php` を手動で作成し、`php.ini` の `opcache.preload` にそのパスを設定します（`opcache.preload_user` の設定も必要です）。サーバー起動時にカーネルコンテナをメモリに事前ロードでき、初回リクエストの速度が大幅に改善します。なお、EC-CUBE は Symfony Flex の自動生成レシピを採用していないため、このファイルは自動生成されません。

## 施策2：APCu キャッシュを有効化する

EC-CUBE の `prod/framework.yaml` のコメントを外して APCu を有効にすると、Symfony のシステムキャッシュ（ルーティング・設定・翻訳等）がプロセスメモリに乗り、ディスクアクセスを大幅に削減できます。

APCu はサーバーに PHP 拡張（`php-apcu`）のインストールが必要です。共有ホスティングでは利用できない場合があります。

---

## 施策3：Doctrine クエリキャッシュを活用する

EC-CUBE の `prod/doctrine.yaml` には、メタデータ・クエリ・結果・セカンドレベルキャッシュの4層すべてが設定済みです。デフォルトは `cache.app` アダプター（ファイルシステム）を使っています。

### Redis・Memcached への切り替え

結果キャッシュを Redis や Memcached に切り替えると、DB への重複クエリを大幅に削減できます。特に商品一覧の絞り込みクエリは実行コストが高いため、効果が出やすい箇所です。

:::message alert
**セキュリティ上の注意：** Doctrine の結果キャッシュは対象クエリを選ばないため、会員の注文履歴・住所・個人情報を含むクエリ結果が Redis や Memcached に保存される場合があります。Redis を利用する場合は `requirepass` によるパスワード認証を必ず設定し、外部ネットワークからアクセスできないようファイアウォールやバインドアドレスで保護してください。個人情報を含むクエリは結果キャッシュの対象から慎重に選定することをお勧めします。
:::

### BaseInfo のセカンドレベルキャッシュ

EC-CUBE の `BaseInfo` エンティティには `@ORM\Cache(usage="NONSTRICT_READ_WRITE")` が設定されており、Doctrine セカンドレベルキャッシュが有効です。セカンドレベルキャッシュが正しく機能していれば、`BaseInfoRepository::get()` の DB アクセスはキャッシュヒット時にスキップされます。

---

## 施策4：Twig テンプレートのウォームアップ

Twig テンプレートは初回アクセス時に `var/cache/{env}/twig` へコンパイルされます。本番環境ではデプロイのたびにキャッシュをクリアするため、**デプロイ直後の初回アクセスが遅くなる**問題があります。

デプロイスクリプトに `php bin/console cache:warmup --env=prod` を組み込むことで、事前コンパイルが完了した状態でサービスインできます。

---

## 施策5：画像の最適化（WebP 対応）

EC-CUBE 4.3 の `.htaccess` には、ブラウザが `Accept: image/webp` を含むリクエストを送った場合に、JPG・PNG を WebP にルーティングする設定が含まれています。サーバーに WebP 変換済みの画像ファイルさえ用意すれば、対応ブラウザには自動的に軽量な WebP が配信されます。

画像最適化の優先度は以下の通りです。

| 施策 | 効果 | 難易度 |
|------|------|--------|
| WebP 変換 | ファイルサイズ 25〜35% 削減 | 低 |
| 画像の遅延読み込み（`loading="lazy"`） | 初期ロード削減 | 低 |
| 適切なリサイズ | 過大な画像の排除 | 低〜中 |
| 次世代フォーマット（AVIF） | さらなる圧縮率 | 高 |

商品画像は EC-CUBE の管理画面からアップロードされますが、**アップロード時に自動リサイズ・WebP 変換を行うプラグインの導入**も選択肢の一つです。

---

## 施策6：HTTP キャッシュ（ESI）の活用

EC-CUBE 4.3 の `framework.yaml` では ESI（Edge Side Includes）が有効化されています（`esi: { enabled: true }`）。ただし、デフォルトのコントローラ実装では ESI キャッシュが実際に活用されているわけではありません。

ESI を活用すると、ページを複数のフラグメントに分割してそれぞれ独立してキャッシュできます。例えば「商品の基本情報（長期キャッシュ）」と「在庫状況・価格（短期キャッシュまたはキャッシュなし）」を分離する設計が可能です。

EC-CUBE には標準のキャッシュ寿命設定として以下が定義されています。

| 種別 | 寿命 |
|------|------|
| 標準キャッシュ | 3600秒（1時間） |
| 短期キャッシュ | 10秒（商品一覧など） |

ESI の本格活用には Varnish などのリバースプロキシとの連携が効果的です。

:::message alert
**セキュリティ上の注意：** Symfony の ESI は `_fragment` エンドポイントを `framework.secret` で署名して保護しています。EC-CUBE では `framework.secret` に相当する値は環境変数 `ECCUBE_AUTH_MAGIC` で管理されています。この値が漏洩すると署名を偽造され、任意のフラグメントを呼び出される恐れがあります。十分な長さのランダム値を使用し、ソースコードにコミットしないようにしてください。また、ユーザー入力を含む文字列を Twig でエスケープせずに出力している箇所がある場合、ESI インジェクションのリスクがあります。Twig のオートエスケープが有効になっていることを確認してください。
:::

## 施策7：CDN の導入

静的ファイル（CSS・JS・画像）を CDN で配信することで、サーバーへのリクエスト数を大幅に削減できます。Cloudflare のような CDN は無料プランでも基本的なキャッシュと DDoS 防御を提供します。

CDN 導入時の注意点：

- EC-CUBE の `Vary: Accept` ヘッダー（WebP 配信用）を CDN が正しく処理するか確認する
- セッションクッキー（`ECCUBE_SESSID`）を持つリクエストはキャッシュ対象外にする
- 管理画面（デフォルトは `/admin`。環境変数 `ECCUBE_ADMIN_ROUTE` で変更している場合はそのパス）は CDN キャッシュから除外する
- マイページ（`/mypage`）・購入フロー（`/shopping`）等の認証必須ページも URL パスベースでキャッシュ除外する（セッションクッキーの判定だけでは CDN の実装によって漏れが生じることがある）
- HTTPS を強制し、HTTP へのアクセスは HTTPS にリダイレクトする設定を CDN 側でも有効にする
- オリジンサーバーへの直接アクセス（CDN バイパス）を防ぐため、オリジンのファイアウォールで CDN の IP レンジ以外からのアクセスを遮断することを検討する

## 施策の優先度まとめ

| 優先度 | 施策 | 難易度 | 費用 |
|--------|------|--------|------|
| 高 | OPcache 設定の最適化 | 低 | 無料 |
| 高 | APCu の有効化 | 低 | 無料 |
| 高 | 画像の WebP 変換 | 低 | 無料〜 |
| 中 | Doctrine 結果キャッシュを Redis に切り替え | 中 | Redis サーバー費用 |
| 中 | デプロイ時の cache:warmup 組み込み | 低 | 無料 |
| 中 | CDN 導入 | 中 | 無料〜 |
| 低 | ESI による部分キャッシュ | 高 | Varnish 等の費用 |

まずは **OPcache・APCu・画像最適化** の3つに取り組むだけでも、体感速度は大きく変わります。費用をかけずにできることから始め、効果を測定しながら次の施策へ進むことをお勧めします。
