**「毎月、誰が誕生月か調べてクーポンを送っている」——そのオペレーション、今月で終わりにできます。**

EC-CUBE 4.3 には、誕生日を検知するイベントが存在しません。多くの開発者がこの事実に気づかず、イベントリスナーでの実装を試みて行き詰まります。

**結論：Symfony の Console コマンドと cron を組み合わせるだけで実現できます。**

公式クーポンプラグインは、EC-CUBE の標準機能と非常に相性よく設計されており、少ない実装量で誕生日クーポンの自動配布が完成します。この記事ではその仕組みと設計の考え方を解説します。

## なぜ「イベントリスナー」では実装できないのか

EC-CUBE は Symfony のイベントシステムをベースに、購入完了・会員登録・受注ステータス変更など多くの場面でイベントを発火します。しかし **誕生日に関連するイベントは定義されていません**。

これは理にかなっています。誕生日は「何かの操作」に紐付いているわけではなく、毎日 DB をチェックして初めて判断できる状態変化だからです。

そのため、正しい設計は次のようになります。

**毎日 cron が定時実行 → Console コマンドが起動 → 当日（または当月）が誕生日の会員を取得 → 個人クーポンを生成してメールで送信**

EC-CUBE は Symfony をベースにしているため、**Symfony の Console コマンド機能をそのままプラグイン内で使えます**。サードパーティのバッチフレームワークは不要で、1つのコマンドクラスを書いて cron に登録するだけです。

## EC-CUBE が標準で持つ誕生日データ

EC-CUBE 4.3 の `Customer` エンティティ（`dtb_customer` テーブル）には、`birth`（生年月日）フィールドが標準で存在します。会員登録時に任意入力してもらうものです。

会員一覧の検索機能も「誕生月での絞り込み」に標準で対応しており、管理画面からも手動で誕生月の会員を抽出できます。Console コマンドでは、この同じ仕組みを使って**今月（または今日）が誕生日の会員を自動取得**します。

## 公式クーポンプラグインとの組み合わせ

公式クーポンプラグイン（EC-CUBEオーナーズストアで公開されている `Coupon42`）は、コードを知っていれば誰でも使える設計です。一見すると「特定の顧客だけに配る」ことには不向きに見えますが、次の工夫で個人クーポンとして機能させられます。

**顧客ごとにランダムなユニークコードを生成 + 使用回数を1回に設定 + 会員限定フラグを有効化**

この3点を組み合わせると、事実上「その会員専用の1回限りクーポン」になります。クーポンプラグイン自体にこのコードを生成するユーティリティが内蔵されているため、複雑な実装は不要です。

| 設定項目 | 値 | 効果 |
|---|---|---|
| クーポンコード | ランダム12文字 | 他人には推測できない |
| 残使用回数 | 1回 | 1度使ったら無効になる |
| 会員限定 | オン | ゲストでは使えない |
| 有効期限 | 当月末まで | 誕生月中にだけ使える |

## 全体の処理フロー

実装する Console コマンドは、次の流れで動作します。

1. **誕生日会員の取得**: 今月（または今日）が誕生日の本会員を DB から取得する
2. **クーポンの生成**: 会員ごとにユニークなクーポンコードを生成して DB に保存する
3. **メール送信**: クーポンコードを記載したお祝いメールを会員のアドレスへ送る
4. **cron で毎月1日（または毎日）自動実行**: サーバーのスケジューラに1行登録するだけ

EC-CUBE のショップ名・メールアドレスは管理画面の「基本情報」で設定した値を自動取得するため、ハードコードは不要です。

## 重複送信を防ぐ設計

cron の誤設定やコマンドの重複実行で同一会員に2通送ってしまうと、ブランドの信頼を損ねます。これを防ぐには**送信履歴テーブルを独自に持つ**のが確実です。

プラグイン内に「誰に・いつ・どのクーポンを送ったか」を記録するテーブルを追加し、コマンド実行前に「今年すでに送ったか」を確認する仕組みを入れます。同じ年に同じ会員に2度送れないよう制約を設けることで、何度コマンドを実行しても安全に動作します。

## 「月初に当月全員へ送る」vs「毎日当日の人へ送る」

どちらの設計も成立しますが、それぞれにトレードオフがあります。

| 方式 | メリット | デメリット |
|---|---|---|
| 月初に当月全員へ送る | cron が月1回で済む | 誕生日の前日にクーポンが届く場合がある |
| 毎日当日の人へ送る | 誕生日当日の朝に届いて感動が大きい | cron が毎日実行される |

誕生日当日の朝に届いたほうが喜ばれやすいため、毎日実行方式が施策としての効果は高くなる傾向があります。

## EC-CUBE だからこそ簡単に実現できる

この機能が EC-CUBE で比較的容易に実装できる理由は、以下の要素が最初から揃っているからです。

- **`dtb_customer.birth`** — 誕生日フィールドが標準テーブルに存在する
- **Symfony Console** — バッチ処理の仕組みがフレームワークレベルで提供されている
- **クーポンプラグイン** — クーポン生成・適用の複雑なロジックが実装済み
- **Doctrine ORM** — 誕生月・誕生日での会員絞り込みクエリが数行で書ける
- **Symfony Mailer** — トランザクションメール送信が標準で使える

スクラッチ開発では会員管理・クーポン管理・メール送信をゼロから作る必要がありますが、EC-CUBE ではこれらがすべて揃っているため、**開発リソースを「施策の設計」と「つなぎ合わせる部分」だけに集中できます**。

## まとめ

- EC-CUBE には誕生日イベントが存在しないため、**Console コマンド + cron** が正しい実装方針です
- 公式クーポンプラグインは「ユニークコード + 使用回数1回 + 会員限定」の組み合わせで、実質的な個人クーポンとして機能します
- 送信履歴テーブルを持つことで重複送信を防ぎ、何度実行しても安全な設計になります
- EC-CUBE は誕生日クーポン自動化に必要な部品が全て揃っており、比較的少ない実装量で実現できます
