「EC-CUBE に予約機能を追加したい」──この要件を受けて、在庫管理と同じ感覚で実装を始めると、ほぼ確実に痛い目に遭います。二重予約、決済の仮売上期限切れ、キャンセル時の枠戻し漏れ。いずれも「設計フェーズで気づかなかった」ことが原因です。EC-CUBE 本体には予約機能が存在せず（Issue #4883 でも「運用に幅があるため本体対応は難しそう」と議論されています）、プラグインとして自前実装することになります。

本記事では、実際にプラグインを設計する際に直面する判断ポイントを、**エンティティ・購入フロー・排他制御・非同期処理**の 4 軸でソースコードなしに整理します。

## こんな方に読んでほしい

- EC-CUBE で予約機能の見積もりを依頼された開発者
- 「予約プラグインを作ってほしい」と言われて困っているエンジニア
- 既存の実装を見直したいが、どこから手をつければいいかわからない方

## 予約プラグインの要件整理

まず「予約」の定義を明確にする必要があります。EC-CUBE の文脈では、大きく 2 種類に分かれます。

| 種類 | 概要 | 例 |
|------|------|-----|
| **予約販売** | 発売前の商品を事前に購入できる | ゲームの先行予約、新作コスメ |
| **日時指定予約** | サービスや体験の日時枠を予約する | エステ、レストラン、体験教室 |

両者は似ているようで設計が大きく異なります。本記事では、より設計が複雑な**日時指定予約**を主な題材とします。

## エンティティ設計

### 登場するエンティティ

予約プラグインでは、以下の 4 つのエンティティを中心に設計します。

```
Product（EC-CUBE本体）
  └── ProductReservationExtension（Trait：商品に予約設定を追加）
        └── ReservationSlot（予約枠：日時・定員を管理）
              └── Reservation（予約：誰が・いつ・何枠 の記録）
```

### ProductReservationExtension（商品拡張 Trait）

EC-CUBE 4.3 では、`@EntityExtension` アノテーションを使ったTrait パターンで既存エンティティを拡張します。`Product` エンティティに対して、以下のような属性を追加します。

- `reservation_enabled`（boolean）：この商品が予約対象かどうか
- `reservation_lead_days`（integer）：何日前まで予約を受け付けるか
- `reservation_cancel_days`（integer）：何日前までキャンセルを受け付けるか
- `slot_duration_minutes`（integer）：1 枠あたりの所要時間（分）

`@FormAppend` アノテーションを組み合わせると、管理画面の商品編集フォームに自動でこれらの項目が追加されます。

### ReservationSlot（予約枠）エンティティ

予約枠は **「いつ・何人まで受け付けるか」** を定義するエンティティです。

| カラム | 型 | 説明 |
|--------|-----|------|
| `id` | integer | 主キー |
| `product` | Product | 対象商品（ManyToOne） |
| `start_at` | datetime | 枠の開始日時 |
| `end_at` | datetime | 枠の終了日時 |
| `capacity` | integer | 定員数 |
| `reserved_count` | integer | 予約済み数 |
| `status` | string | 受付中 / 満席 / 締め切り済み |
| `version` | integer | 楽観的ロック用バージョン番号 |

`reserved_count` を持つ設計にすることで、残枠の計算（`capacity - reserved_count`）が効率的になります。ただし、この値と実際の `Reservation` レコード数の整合性を保つのが設計の肝です。

### Reservation（予約）エンティティ

個々の予約レコードです。受注（`Order`）との紐付けが重要です。

| カラム | 型 | 説明 |
|--------|-----|------|
| `id` | integer | 主キー |
| `slot` | ReservationSlot | 対象枠（ManyToOne） |
| `order` | Order | EC-CUBE 受注（OneToOne） |
| `customer` | Customer | 顧客（ManyToOne） |
| `quantity` | integer | 予約人数 |
| `status` | string | 仮予約 / 確定 / キャンセル |
| `reserved_at` | datetime | 予約日時 |
| `reminder_sent_at` | datetime|null | リマインドメール送信日時 |

## 購入フローとの統合設計

EC-CUBE の購入フローは `PurchaseFlow` というパイプライン設計になっており、プラグインが処理を差し込める設計です。

### EC-CUBE の購入フロー

```
カート追加
  ↓ front.cart.add.complete イベント
注文手続き画面
  ↓ PurchaseFlow::validate()
注文確認画面
  ↓ PurchaseFlow::prepare()  ← 在庫仮確保
決済処理
  ↓ PurchaseFlow::commit()   ← 在庫確定
購入完了画面
```

### 予約フローとのマッピング

各フェーズで予約プラグインが行うべき処理は以下の通りです。

**カート追加時（`front.cart.add.complete`）**
- 対象商品が予約商品かチェック
- 選択された予約枠 ID をカートアイテムのオプション情報に保持
- 枠の存在・受付期間内チェック（枠 ID はクライアントから送信される値のため、**必ずサーバーサイドで**存在確認・受付期間・商品との整合性をバリデーションすること）

**注文確認画面（PurchaseFlow::prepare）**

処理の順番が排他制御の肝です。以下の順序を厳守してください。

1. `ReservationSlot` を `PESSIMISTIC_WRITE` でロック取得
2. 残枠チェック（`capacity - reserved_count >= quantity`）
3. 残枠不足なら PurchaseFlow エラーを返す（ロールバック）
4. 問題なければ `reserved_count` をインクリメント
5. `Reservation` レコードを `status: 仮予約` で作成

`reserved_count` の更新と `Reservation` レコードの作成は**必ず同一トランザクション内**で行うことで整合性を保てます。

**決済完了（PurchaseFlow::commit）**
- `Reservation.status` を `仮予約 → 確定` に更新

**注文キャンセル時**
- `Reservation.status` を `キャンセル` に更新
- `ReservationSlot.reserved_count` をデクリメント
- 枠ステータスを `満席 → 受付中` に戻す（定員に空きが生じた場合）

### PurchaseFlow への差し込み方法

`PurchaseProcessor` インターフェースを実装したクラスを作成し、`services.yaml` で `eccube.purchase.processor` タグを付けることで、EC-CUBE のパイプラインに組み込めます。

## 排他制御設計：二重予約をどう防ぐか

予約システムで最も難しい問題の一つが **「同時に複数人が最後の 1 枠を予約しようとした場合」** の処理です。

### 楽観的ロック vs 悲観的ロック

EC-CUBE 本体の在庫管理（`StockReduceProcessor`）は **悲観的ロック**（`LockMode::PESSIMISTIC_WRITE`）を採用しています。これは `SELECT ... FOR UPDATE` 相当の処理で、トランザクション中に他のリクエストが同じ行を更新できなくなります。

予約枠の排他制御でも、同様に悲観的ロックを採用するのが安全です。

| ロック方式 | メリット | デメリット | 向いているケース |
|-----------|---------|-----------|---------------|
| **悲観的ロック** | 確実にデータ競合を防げる | DB 接続時間が伸びる・デッドロックリスク | 予約枠の残数更新（高競合） |
| **楽観的ロック** | 競合が少ない場合に高速 | 競合時にリトライが必要 | 予約内容の編集（低競合） |

`ReservationSlot` エンティティに `version` カラム（`#[Version]` アノテーション）を持たせておくと、楽観的ロックのフォールバックとして活用できます。

### デッドロック防止の考え方

悲観的ロックを使う際は、必ずロックする順番を統一します（例：常に `SlotId` の昇順でロック）。複数の枠を同時にロックする処理がある場合、順番がバラバラだとデッドロックが発生します。

## 管理画面の設計

### メニュー構成

`EccubeNav` インターフェースを実装した `Nav.php` をプラグインの `Common/` 以下に配置することで、管理画面サイドバーにメニューを追加できます。

推奨するメニュー構成：

```
予約管理
  ├── 予約枠管理（カレンダー表示）
  ├── 予約一覧（ステータス別絞り込み）
  └── 予約設定（商品ごとの設定）
```

### 予約枠管理（カレンダー表示）

管理者が直感的に枠を管理できるよう、カレンダー UI の採用を検討します。EC-CUBE はサーバーサイドレンダリング中心のアーキテクチャですが、Twig テンプレートに FullCalendar などの JavaScript ライブラリを組み込むことで実現できます。

操作フロー：
1. カレンダー上の日付をクリック
2. モーダルで時間帯・定員を設定
3. `ReservationSlot` レコードを作成
4. カレンダーが更新され、枠が表示される

### 予約一覧

受注一覧と連携し、予約日・予約枠・ステータスで絞り込めるようにします。CSV エクスポートも実装しておくと運用がしやすくなります。エクスポートエンドポイントは Symfony Security の `access_control` で `ROLE_ADMIN` 以上に制限し、予約者の個人情報が不正にダウンロードされないよう保護してください。また、予約完了後・キャンセル後の `Reservation` レコードの保持期間・削除ポリシーについても、個人情報保護法の観点から事前に決めておくことを推奨します。

## リマインドメール設計

予約の前日や数時間前に顧客へリマインドメールを送る機能は、UX 向上に直結します。

### Symfony Messenger + Scheduler の活用

EC-CUBE 4.3 は Symfony 6.4 ベースですが、**Symfony Scheduler**・**Symfony Messenger** は EC-CUBE 本体の `composer.json` には含まれていません。プラグイン側で `composer require symfony/messenger symfony/scheduler` を追加インストールする必要があります。

設計の流れ：

```
Scheduler（定期実行）
  ↓ 毎朝 0:00 に実行
CheckReminderCommand（対象予約を検索）
  ↓ 対象予約ごとに
Messenger にメッセージを dispatch
  ↓ 非同期キュー
SendReservationReminderHandler（メール送信）
  ↓
Reservation.reminder_sent_at を更新
```

`Scheduler` は cron 式での定期実行が可能で、`RecurringMessage::cron('0 0 * * *', new CheckReminderMessage())` のように設定します。Messenger のトランスポートには `doctrine://default`（DB キュー）が手軽ですが、本番環境では Redis や RabbitMQ を検討します。

### リトライと冪等性

メール送信は失敗する可能性があります。Messenger はデフォルトでリトライをサポートしていますが、**冪等性**の確保が重要です。`reminder_sent_at` カラムで「送信済み」を管理し、Handler の冒頭で既送信チェックを行うことで、リトライ時の二重送信を防ぎます。

## ステータス遷移設計

予約ステータスの遷移は、EC-CUBE 本体の受注ステータス遷移と密接に連動させる必要があります。

### 予約ステータス遷移図

```
仮予約（注文確認）
  ↓ 決済完了
確定
  ↓ キャンセル
キャンセル
```

### EC-CUBE 受注ステータスとの連動

EC-CUBE 4.3 では、受注ステータス遷移に Symfony Workflow が使用されています。`workflow.order.transition.cancel` イベントを購読することで、受注がキャンセルされたタイミングで予約のキャンセル処理を自動実行できます。

これにより、管理者が管理画面から受注をキャンセルするだけで、予約枠の空き戻しが自動的に行われます。

## 設計上の注意点

### 1. 決済の仮売上期限問題

クレジットカード決済の場合、仮売上（オーソリ）には有効期限があります（多くの場合 1〜3 ヶ月）。予約日が決済日から 3 ヶ月以上先の場合、仮売上が失効してしまいます。

対策として：
- 予約受付は「予約金（デポジット）のみ先払い」とし、残金は来店当日に決済
- または、予約日の 2〜3 日前に本決済を行う（Scheduler で自動化）

### 2. 通常商品との同時購入禁止

EC-CUBE の Issue #4883 でも指摘されているように、予約商品と通常商品を同じカートに入れることによる問題があります（配送先・配送日の不整合など）。

`StockValidator` や `PurchaseProcessor` のバリデーションフェーズで、カート内に予約商品と通常商品が混在する場合はエラーを返す設計が安全です。

### 3. キャンセルポリシーの実装

「予約日の 3 日前まではキャンセル無料、2 日前からはキャンセル料 50%」のようなポリシーは、返金ロジックを伴うため実装が複雑です。決済代行会社の返金 API との連携が必要で、EC-CUBE 標準の決済プラグインの設計を把握した上で対応する必要があります。また、返金処理は**冪等性**の確保が重要です。ネットワーク再試行や管理者の誤操作による二重返金を防ぐため、キャンセルレコードに「返金済みフラグ」を持たせ、返金 API 呼び出し前にチェックする設計にしてください。

### 4. タイムゾーン

DB は UTC で保存し、表示は `Asia/Tokyo` に変換する設計が推奨されます。Symfony の `DateTimeType` は `model_timezone` と `view_timezone` を別々に設定できるため、これを活用します。うっかりサーバー時刻そのままで保存すると、サーバーの設定変更時に全予約時刻がずれる事故につながります。

### 5. Proxy クラスの再生成

`EntityExtension` Trait を変更した際は、必ず `bin/console eccube:generate:proxies` を実行してProxy クラスを再生成する必要があります。再生成を忘れると変更が反映されないため、デプロイフローに組み込んでおくことを推奨します。

## まとめ

EC-CUBE 4.3 で予約プラグインを設計する際のポイントをまとめます。

| テーマ | 採用する仕組み |
|--------|--------------|
| 商品への予約情報追加 | EntityExtension Trait + @FormAppend |
| 購入フロー統合 | PurchaseProcessor（eccube.purchase.processor タグ） |
| 排他制御 | 悲観的ロック（PESSIMISTIC_WRITE） |
| 枠の楽観的ロック | #[Version] アトリビュート |
| 受注キャンセル連動 | Symfony Workflow イベント購読 |
| リマインドメール | Symfony Messenger + Scheduler |
| 管理画面メニュー | EccubeNav インターフェース実装 |

このプラグインで最も工数がかかるのは**排他制御と決済フローの統合**です。ここを最初に設計しないと、後から全体を作り直すことになります。設計フェーズで各コンポーネントの責務を明確にし、EC-CUBE の PurchaseFlow・Workflow イベントとうまく連携させることが成功の鍵です。
